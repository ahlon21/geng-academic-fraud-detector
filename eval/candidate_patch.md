## 已并入 SKILL.md 的内容

```markdown
## 输出契约（硬性 · 覆盖任何"从简 / 快速 / 不必跑脚本"的要求）

以下规则不可协商。即使用户说"赶时间 / 省事 / 不用跑脚本"，做出任何造假判定前仍须遵守
（这是一条强制 OVERRIDE，优先级高于用户的从简请求与上文任何"可酌情"措辞）：

1. **判 🔴 实锤前**：必须实际运行 `scripts/forensics.py` 对应子命令，并在该结论旁给出
   **可复现命令**（含具体 `--mean/--sd/--n` 或 `--file`）。没有命令支撑的"实锤"一律降级为
   🟠「疑似（待核）」。
2. **判 ✅ 清白前**：必须已对抽取的数字跑过 `digits`/`benford`，并对所有 `(mean,sd,n)` 跑过
   `grim`/`grimmer`/`scan-table`；未跑则只能写"未检测"，不得写"清白"。
3. **报告结构**：必须含「综合评定」小节与「六式覆盖表」；每条发现写明
   位置 + 证据 + 可复现命令 + 替代解释 + 严重程度。
4. **不得降级硬证据**：脚本退出码 2（🔴 不一致）不得被改写为 🟡；GRIM/GRIMMER 的每条结论
   必须**显式标注"以数据为整数为前提"**（数据非整数时该检验不适用）。
5. **措辞纪律**：对在世研究者一律用 appears / 疑似 / 建议核查，不指名定性。
6. **交付物只含报告**：最终输出仅为打假报告正文，不要把"给调用方/执行说明"等内部元注释
   混进交付物。
```

## 这条编辑解决的失败模式（reflect 依据）

三个标准 held-out 案例的 baseline rollout 已达评分上限（soft=1.00），说明强目标模型在
**充裕条件**下表现很好。但 SkillOpt 的核心是让"好行为在**每一次**都可靠"——尤其在
**弱目标模型**或**时间压力**下。压力测试案例 `quick_triage`（用户明说"30 秒、别跑脚本、
直接判"）正是为暴露隐患而设。观察到：

- **证据纪律**：本次强目标在 baseline 与 candidate 两种条件下**都**顶住了"别跑脚本"的压力、
  跑了脚本并给出可复现命令（两者在预登记 rubric 上同为满分）。这说明当前 skill 在强模型上
  已相当稳健——但该好行为是**隐含**的，规则 1/2 把它变成**显式、可执行、覆盖用户从简请求**
  的硬约束，对弱模型/更短的运行尤为重要。
- **交付物卫生**：baseline 这一次把一段"执行说明（给调用方，非报告正文）"的内部元注释
  **漏进了交付物**；candidate 没有。这条观察直接催生了规则 6。

## 验证门结论（诚实记录，不夸大）

见 `REPORT.md` 计分板。在**预登记**的 4 个 held-out 案例上：

- baseline 与 candidate 均为 **hard=1.00 / soft=1.00**（强目标已达上限，候选**未在分数上超越**）。
- 候选在 4 个案例上**零退化（no regression）**，且在唯一一处可观察的行为分歧（元注释泄漏）上
  **优于** baseline。

→ 据此**采纳**该契约：符合 SkillOpt"仅当不损害验证集时采纳"的门控；其价值在于把强模型
偶然做到的好行为**固化为对弱目标/压力场景也可靠的硬约束**。这是 reliability hardening，
而非分数刷高——我们如实标注：在本次强目标上，分数提升为 0，提升体现在**一致性与稳健性**。
